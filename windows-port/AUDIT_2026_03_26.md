## SEVERITY: CRITICAL (Will cause runtime failures)

### 1. Registry Path Mismatch: Shim Never Reads Configured Values

**Files**: `shim/greenboost_cuda_shim_win.c:805`, `tools/install.ps1:384`, `tools/config.ps1:1008`

The driver and shim read configuration from **different** registry paths. Nobody writes to the shim's path.

| Component | Registry Path | Reads/Writes |
|-----------|--------------|--------------|
| Driver | `HKLM\SYSTEM\CurrentControlSet\Services\GreenBoost\Parameters` | Reads |
| install.ps1 | `HKLM:\SYSTEM\CurrentControlSet\Services\GreenBoost\Parameters` | Writes |
| config.ps1 | `HKLM:\SYSTEM\CurrentControlSet\Services\GreenBoost\Parameters` | Writes |
| **Shim** | **`HKLM\SOFTWARE\GreenBoost`** | **Reads** |
| diagnose.ps1 | `HKLM:\SOFTWARE\GreenBoost` | Reads |

**Impact**: The shim ALWAYS runs with hardcoded defaults (12GB physical, 51GB virtual, 256MB threshold) regardless of what install.ps1 or config.ps1 configure. On a system with a 5090 (32GB) and 96GB RAM, the shim will still report 63GB combined instead of the configured amount.

**Fix**: Either change shim `gb_read_config()` to read from `SYSTEM\CurrentControlSet\Services\GreenBoost\Parameters`, or have install.ps1 write to BOTH paths.

### 2. Pressure Event Name Mismatch: Shim Can Never Find It

**Files**: `driver/greenboost_win.h:137`, `shim/greenboost_cuda_shim_win.c:1138-1139`, `tests/test_ioctl.c:371`

| Component | Event Name | Namespace |
|-----------|-----------|-----------|
| Driver (kernel) | `\BaseNamedObjects\GreenBoostPressure` | Global kernel namespace |
| test_ioctl.c | `Global\GreenBoostPressure` | Correctly maps to `\BaseNamedObjects\` |
| **Shim** | **`GreenBoostPressure`** | **Session-local namespace** |

The shim calls `OpenEventW(SYNCHRONIZE, FALSE, L"GreenBoostPressure")` without the `Global\` prefix. In user mode, this resolves to `\Sessions\<N>\BaseNamedObjects\GreenBoostPressure`, which is a completely different object from what the driver created.

**Impact**: The shim never receives pressure notifications. Under memory pressure, the driver signals the event but the shim doesn't know. The shim continues allocating while the system is OOM. The watchdog's entire pressure signaling system is disconnected from the consumer.

**Fix**: Change shim line 1139 to `L"Global\\GreenBoostPressure"`.

---

### 3. No Process Crash Cleanup (Buffer Leak on Process Death)

**Files**: `driver/greenboost_win.c` (GbEvtDeviceAdd)

The driver registers no `EvtFileCleanup` or `EvtFileClose` callback. When a process opens `\\.\GreenBoost`, allocates buffers, then crashes (or is killed), those buffers remain allocated forever until driver unload. The user-mapped VA becomes stale, the MDL pages stay pinned, and pool accounting is never decremented.

The code documents this at line 891: "Windows has no equivalent automatic cleanup for MDL mappings." But the correct fix exists: track buffer ownership per WDFFILEOBJECT and clean up in EvtFileCleanup.

**Impact**: Any shim crash (or parent process kill) permanently leaks pinned physical memory. On a system allocating 40+GB of model weights, one crash means 40GB of RAM is gone until reboot or manual driver reload.

**Fix**: Add WDF file object context, track allocated buf_ids per file object, free all in EvtFileCleanup.

---

### 4. Eviction is Fake (Relabels Only, Memory Never Moves)

**Files**: `driver/greenboost_win.c:1050-1064`

`GbHandleEvict` changes `buf->Tier` from `GB_TIER2_DDR4` to `GB_TIER3_NVME` and adjusts accounting. But the actual memory stays pinned in non-paged pool. No pages are freed or moved to pagefile.

**Impact**: After "eviction," T2 accounting shows freed capacity, allowing more T2 allocations. But the physical RAM is still consumed. The safety reserve math becomes wrong. The system can exhaust RAM while accounting says there's headroom.

If the shim uses MADVISE_COLD + EVICT to manage layer cycling (common in LLM inference), it thinks it freed memory, allocates more, and the OOM killer fires.

## SEVERITY: HIGH (Functional gaps that will bite in production)

### 5. IAT Hooking Fallback Only Hooks cudaMalloc

**Files**: `shim/greenboost_cuda_shim_win.c:1066-1091`

Without Detours (the default unless explicitly installed via vcpkg), only `cudaMalloc` gets IAT-hooked in the main executable. Not hooked:
- `cudaFree` (leaked buffers, every single one)
- `cuMemAlloc_v2` / `cuMemFree_v2` (driver API, used by many frameworks)
- `cudaMemGetInfo` / `cuMemGetInfo` (no VRAM spoofing)
- `cuDeviceTotalMem_v2` (no total VRAM spoofing)
- NVML hooks (monitoring tools see real VRAM)
- Async variants

Also, IAT hooking only patches the main executable's import table. DLLs loaded by the app (like llama.cpp's .dll or Python extension modules) have their own IATs that are never patched.

**Impact**: Without Detours, the shim is effectively broken. It intercepts allocations but can never free them, and VRAM reporting is unmodified so the app may not even try to use the extended memory.

---

### 6. GbHandleReset is a No-Op

**Files**: `driver/greenboost_win.c:978-983`

```c
NTSTATUS GbHandleReset(_In_ WDFREQUEST Request)
{
    UNREFERENCED_PARAMETER(Request);
    gb_info("RESET requested -- close all handles to release buffers");
    return STATUS_SUCCESS;
}
```

Logs a message and returns success without doing anything. The comment says "close all handles to release buffers" but the code doesn't.

**Impact**: Any tool or shim calling RESET to clean up state gets told "success" but nothing changes. Pool stays allocated.

---

### 7. cudaMallocManaged (Runtime API) Not Intercepted

**Files**: `shim/greenboost_cuda_shim_win.c:940-969`

The Detours hook list hooks `cuMemAllocManaged` (driver API) indirectly through the UVM probe, but never hooks the runtime API `cudaMallocManaged`. The real function pointer `real_cudaMallocManaged` is resolved but never passed to `DetourAttach`.

Any framework calling `cudaMallocManaged` (PyTorch's managed memory path, some ONNX Runtime configs) bypasses GreenBoost entirely.

---

### 8. build.ps1 Doesn't Collect Driver Artifacts

**Files**: `windows-port/build.ps1:296-336`

The driver build is disabled (`GB_BUILD_DRIVER=OFF` in CMake config, line 151 has it commented out). Steps 4/5 (signing, catalog) are entirely commented out. Step 6 only collects the shim DLL and test exe.

`install.ps1` expects `greenboost_win.sys` and `greenboost_win.inf` in `tools/outputs/`, but build.ps1 never puts them there.

**Impact**: Running build.ps1 then install.ps1 will fail at driver installation with "Driver SYS not found."

## SEVERITY: MEDIUM (Correctness/robustness issues)

### 9. SDDL Grants World Read/Write Access

**Files**: `driver/greenboost_win.c:1291-1292`

```c
DECLARE_CONST_UNICODE_STRING(sddl,
    L"D:P(A;;GA;;;SY)(A;;GA;;;BA)(A;;GRGW;;;WD)");
```

`WD` = Everyone (World). Any unprivileged process can open the device and issue IOCTLs, including ALLOC (pins arbitrary amounts of physical RAM) and PIN_USER_PTR (maps arbitrary user pages into kernel tracking).

For a personal dev driver this is fine. For distribution, this is a privilege escalation / DoS vector.

---

### 10. Memory Spoofing Can Underflow

**Files**: `shim/greenboost_cuda_shim_win.c:721-723`

```c
*free = orig_free + (gb_config.ReportedTotalVram - orig_total);
```

If `ReportedTotalVram < orig_total` (misconfigured, or running on a GPU with more VRAM than the config specifies), this performs unsigned subtraction that wraps to ~2^64, reporting petabytes of free memory.

**Fix**: Add `if (gb_config.ReportedTotalVram > orig_total)` guard.

---

### 11. INF KmdfLibraryVersion Placeholder

**Files**: `driver/greenboost_win.inf:74`

```ini
KmdfLibraryVersion = $KMDFVERSION$
```

This placeholder must be substituted by the WDK build toolchain (stampinf.exe). If the INF is used as-is (e.g., someone copies it manually), driver installation will fail with a cryptic version mismatch error.

---

### 12. Shim DeviceHandle Initialized to 0 (Valid Handle)

**Files**: `shim/greenboost_cuda_shim_win.c:41`

`gb_shim_config_t gb_config = { 0 }` initializes `DeviceHandle` to 0. The constant 0 is a valid Windows handle (stdin). `gb_read_config()` corrects it to `INVALID_HANDLE_VALUE` (-1) before anything uses it. But any code path that touches the device before `gb_read_config` completes would use handle 0.

Currently theoretical (init order is correct), but fragile.

---

### 13. Hash Table Lacks Rehashing/Load Factor Management

**Files**: `shim/greenboost_cuda_shim_win.h:575-577`, `shim/greenboost_cuda_shim_win.c:174-198`

131K slots, open-addressed with tombstones. No rehashing, no load factor check, no tombstone reclamation. Long-lived processes with many alloc/free cycles will accumulate tombstones, degrading probe chain length toward O(n). At 50%+ tombstone fill, every lookup probes hundreds of slots under lock.

For LLM inference (allocate weights once, keep forever), this is fine. For dynamic workloads, it'll degrade.

## Summary

| Severity | Count | Key Issues |
|----------|-------|-----------|
| CRITICAL | 4 | Registry mismatch, event name mismatch, no crash cleanup, fake eviction |
| HIGH | 4 | IAT-only path broken, RESET no-op, cudaMallocManaged unhooked, build pipeline broken |
| MEDIUM | 5 | SDDL too open, spoofing underflow, INF placeholder, handle init, hash table degradation |
| LOW | 7 | Committed artifacts, pip logs, Chinese script, ASCII art, Ollama reference, missing test target, step numbering |

The driver kernel code is well-written and the Linux port translation is careful (correct MDL usage, proper SEH, good contiguous/non-contiguous fallback). The main problems are in the **integration seams**: the shim and driver don't agree on registry paths or event names, the build pipeline is half-commented-out, and there's no crash safety for buffer cleanup.
